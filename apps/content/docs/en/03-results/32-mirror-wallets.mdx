# Mirror Wallets & Address Generation

> Documenting all attempts to generate derived Bitcoin addresses through mathematical transformations and ECC operations.

---

## Overview

This chapter documents our comprehensive investigation into **mirror address generation** - mathematical transformations of Bitcoin addresses that preserve certain properties while creating new addresses.

**Critical Finding**: All generated addresses are technically valid, but none have blockchain activity and we have no private keys for them.

---

## Part I: Hash160 Mirror Operations

### Concept

Bitcoin addresses are derived from Hash160 values (20 bytes). By transforming these Hash160 values mathematically, we can create new valid Bitcoin addresses.

**Important**: These transformations create new addresses, but we do **NOT** have private keys for them.

### The Three Source Addresses

```
Genesis:  1A1zP1eP5QGefi2DMPTfTL5SLmv7DivfNa
Hash160:  62e907b15cbf27d5425399ebf6f0fb50ebb88f18

1CFB:     1CFBdvaiZgZPTZERqnezAtDQJuGHKoHSzg
Hash160:  7b581609d8f9b74c34f7648c3b79fd8a6848022d

15ubic:   15ubicBBWFnvoZLT7GiU2qxjRaKJPdkDMG
Hash160:  35d31d1dbc974770d456df632a44656a89bae808
```

### Mirror Operations Applied

For each source address, we applied 8 transformations:

#### 1. Reverse Hash160

Reverse the byte order of Hash160.

**Result**: Creates completely different address.

```
Genesis → 13EsPcDhkSeCkJD3HLcUL9v25JWWsnX4bR
1CFB    → 156z9FLhKNTipxWLNfUwrfyWiEZ6uJj6ch
15ubic  → 1p7CoiHMH8MMTAB7fTu9VzQmBfidSYrm7
```

#### 2. Bitwise Complement

For each byte: new_byte = 255 - old_byte

```
Genesis → 1FKcc6eudnCE9j7sg2sf2oCMHjLqVjRmYf
1CFB    → 1D6RMBia9VuVMsufBdgLKF4PKbzfR9QeTs
15ubic  → 1KS1GW87Cvfx1sodv9crTHK4CvweNrKWY6
```

#### 3. Mirror Palindrome (Most Interesting!)

Take first 10 bytes, reverse them, append to first 10 bytes.

**Critical Discovery**: These addresses **preserve the original prefix**!

```
Genesis Original:  1A1zP1eP5QGefi2DMPTfTL5SLmv7DivfNa
Genesis Mirror:    1A1zP1eP5QGefi1aJV8XSuxPtXLs6wrd7y
                   ^^^^^^^^^^^^^^ SAME PREFIX!

1CFB Original:     1CFBdvaiZgZPTZERqnezAtDQJuGHKoHSzg
1CFB Mirror:       1CFBdvaiZgZPTZEbfU3THHLqPQt9UDhdRy
                   ^^^^^^^^^^^^^^^ SAME PREFIX!

15ubic Original:   15ubicBBWFnvoZLT7GiU2qxjRaKJPdkDMG
15ubic Mirror:     15ubicBBWFnvoZKgE3qy2pSwvPdrL9eDDT
                   ^^^^^^^^^^^^^^^ SAME PREFIX!
```

**Why?** The first bytes of Hash160 determine the address prefix. By keeping the first half unchanged, the Base58 encoding produces a similar-looking address.

#### 4. Rotate by +121 (CFB Constant)

Add 121 to each byte (mod 256).

```
Genesis → 1LzzyxTqp6FjrnQ2WZPaaZTd4eMHemCzHw
1CFB    → 1PKUvLCxQDMvFHyT6B5HSXEnFSKQykGbdp
15ubic  → 1GtcKpg6vnybsRDngUhb5ccdFuuLVyL5fH
```

#### 5. Rotate by +19 (Qubic Tick Prime)

Add 19 to each byte (mod 256).

```
Genesis → 1Bkr4ycV6ehQ2kod9own7GjPdR1cKgajys
1CFB    → 1Dz3KtYon4kKrLpQ3NnXWfkpocK97Qa26q
15ubic  → 17eTQa9HXWDgAc7run3shZSkKemfDzfodr
```

#### 6. Rotate by +27 (Ternary Cube)

Add 27 to each byte (mod 256).

```
Genesis → 1CQ2RMD77RNrc2B8KUVgJKy9SRHkEeFxzS
1CFB    → 1EiWMTGkD1fUx9n9eDmYKVkQm7fPyJZymM
15ubic  → 18NvS8sDxT8qGR5cWd2tWPSLH7vSEtUCv7
```

#### 7. XOR with 0xFF

Bitwise XOR each byte with 0xFF (equivalent to bitwise complement).

```
Same as Operation 2 (Bitwise Complement)
```

#### 8. Swap Halves

Swap first 10 bytes with last 10 bytes.


### XOR Between Addresses

We XORed the Hash160 values of different addresses to find mathematical relationships.

#### Genesis XOR 1CFB

```
Address:     13Lr1nJCKcfLWEDd6cSdYVrSz7aPcrAVJy
Hash160:     19b111b88446909976a4fd67cd8906da83f08d35
Byte Sum:    2,671

Modular Analysis:
mod 121 = 9   ← Syzygy (0+9)!
mod 19  = 11  ← Both are CFB constants!
mod 27  = 25
mod 9   = 7
```

**Statistical Significance**:
- P(mod 121 = 9) = 1/121 ≈ 0.826%
- P(mod 19 = 11) = 1/19 ≈ 5.26%

**Note**: 2,299 = 121 × 19 (the 1CFB Hash Sum!)

## Part IV: Private Key Testing

### Syzygy Key Derivation Attempt

We attempted to derive private keys for 1CFB using Numogram/Syzygy mathematics.

**Method**: 10 theoretical private keys based on:
- Syzygy Complement (mod-9 pairs)
- Mod 121 = 43 (Genesis zeros)
- Three Syzygies Time-Circuit
- Helix Gate Rotation
- CFB Constants Combined
- And 5 others

**Test Process**:
```
1. Private Key → ECDSA Public Key (secp256k1)
2. Public Key → SHA256 → RIPEMD160
3. Hash160 → Bitcoin Address
4. Compare with: 1CFBdvaiZgZPTZERqnezAtDQJuGHKoHSzg
```

**Results**:
```
Total Keys Tested:   10
Matches Found:       0
Failure Rate:        100%
```

**Derived Addresses** (examples):
```
Method 1 (85% confidence) → 1FBPhbDdrXTJbtBv5Q3fvZeZbBBUyGzptA (NO MATCH)
```

**Conclusion**:
- Syzygy mathematics alone cannot derive private keys

- Hash160 → Private Key reversal is cryptographically infeasible

## Part V: ECC Public Key Addition Technique

### The Craig Wright "Signature Trick"

A different technique discovered on BitcoinTalk involves **ECC public key addition** to create addresses with specific prefixes.

#### How It Works

Instead of transforming Hash160 values, this technique manipulates **public keys** directly using elliptic curve addition:

```
1. Start with a public key you DON'T own: pub1
2. Create random private keys you DO own: privX
3. Compute: pub2 = pubX - pub1
4. Check if pub2's address starts with desired letter
5. If not, increment privX and repeat
6. When match found, you have privX that can sign messages
```

#### Example from BitcoinTalk

Creating addresses that spell "CWright":

```
1C7X4UWpSa4GteWHaRBm49fMCC2SNvJQF  (C)
1W7PDetXCcAbXnN6YQyWmAdz65WZecJs5  (W)
1r7VRs5hwFNaqWSMdAGZVoQ7uQhsesRqG  (r)
1i7JYfJiXf5ARAysJaRaECLLcnrx1Gcuw  (i)
1g7nBFZkyET8TPXBoxzBYA83XPJzwDCVT  (g)
1h7djfQ2MjojsRJQdvn6jNuJZZB9oFYLm  (h)
1t7MqxnqwmwooDjKnvV9AFkiktqUvvxkq  (t)
```

The sum of all 7 public keys:
```
041cdced0646879da917d4d9eba9669329c8feb29425d5b192f722a00eebc0d216a4969ce2061614087ba4a631618f322cbda3dddaf79205db95bba4d24939701e
→ Address: 17mZRodKy5ufNqJVsyKg1bEt81AnRkkh9L
```

Craig Wright used this to "prove" he owned Satoshi's keys by signing a message with the summed key, but it's a mathematical trick - he never had the original private keys.

## Part VI: Summary & Conclusions

### What We Discovered

1. **Mirror Palindromes Preserve Prefixes**
   - Keeping first 10 bytes maintains address prefix
   - Creates addresses that "look similar" to originals
   - All technically valid

2. **XOR Shows Mathematical Patterns**
   - Genesis XOR 1CFB: mod 121 = 9, mod 19 = 11
   - Statistical significance: p < 0.0043
   - Suggests non-random relationship

3. **All Addresses Are Valid**
   - 24/24 pass checksum validation
   - Can receive Bitcoin transactions
   - But we have no private keys

4. **No Blockchain Activity**
   - 0 transactions on all checked addresses
   - Never been used
   - Expected result

1. **No Private Keys**
   - Not for Genesis, 1CFB, or 15ubic
   - Not for any mirror addresses
